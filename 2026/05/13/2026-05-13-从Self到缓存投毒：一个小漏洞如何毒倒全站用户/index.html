从Self到缓存投毒：一个小漏洞如何"毒"倒全站用户

起因：一个Self-XSS漏洞

最近关注到 CVE-2026-44581，Next.js 应用路由器从请求头中提取 CSP nonce 并镜像到内联 <script nonce="..."> 元素上，让浏览器允许脚本执行。但存在安全漏洞——攻击者可以构造恶意 Content-Security-Policy 请求头：

提取出的 nonce 值为 " onerror="alert(1)，最终渲染为：

浏览器解析 nonce=""（空值），然后将其识别为独立的 onerror="alert(1)" 属性。当脚本元素处理失败时，onerror 触发 → 受害者源中的任意 JavaScript 执行，XSS 诞生。

参考：https://github.com/dwisiswant0/next-16.2.4-pocs/tree/base/poc/CVE-2026-44581_GHSA-ffhc-5mcf-pf4q

问题：这只是Self-XSS

漏洞确实存在，但这里有个关键问题——这个 XSS 需要攻击者自己构造恶意请求头才能触发。普通用户不会在请求中带上恶意的 Content-Security-Policy 头，所以它本质上是一个请求头反射型 XSS，仅影响发送请求者自身（安全社区常将这类漏洞归类为 Self-XSS），危害有限。

但 Self-XSS 真的就只能”自己打自己”吗？

扩大战果：缓存投毒

如果该响应可被 CDN 缓存，那我能让恶意响应被缓存，其他用户访问同一页面时就会命中投毒缓存——Self-XSS 就升级成了影响全站的存储型 XSS。

关键发现

查看目标站点的响应头，发现 Vary 仅为 Accept-Encoding：

这意味着 CDN 缓存键只包含请求行 + Host + Accept-Encoding，Content-Security-Policy 请求头不参与缓存键计算。也就是说，我带上恶意 CSP 头的请求和正常用户的请求，只要 Accept-Encoding 相同，就会命中同一份缓存。

攻击流程

  1. 构造带恶意 Content-Security-Policy 头的请求访问目标页面
  2. CDN 缓存了包含 onerror="alert(1)" 的投毒响应
  3. 同地区、同 Accept-Encoding 的正常用户访问该页面，命中投毒缓存
  4. 所有这些用户触发 XSS

如何确认投毒成功

投毒请求首次发送时 age 通常为 0（直接回源），正确的做法是：投毒后再发一个正常请求（不带恶意头），如果返回的响应 age > 0 且 x-cache-status: HIT，才说明投毒响应已被缓存且正在被分发。结合 x-swift-cachetime 可以判断缓存还会存活多久，评估攻击窗口。

影响范围

受限于 CDN 缓存节点分布，同 Accept-Encoding、同地区的用户都有可能触发投毒缓存。虽然不是全球所有用户，但对于高流量页面，单一节点覆盖的用户量已经相当可观。

缓存投毒基础知识

上面的实战案例展示了 Self-XSS 如何通过缓存投毒扩大影响。下面系统梳理缓存投毒的原理和必要条件。

什么是Web缓存

网站通常通过 CDN、负载均衡器或反向代理实现 Web 缓存功能，缓存频繁访问的资源（特别是 JS 等静态文件）以降低服务器响应延迟。

缓存键（Cache Key）

缓存键是用来识别等效请求的一系列特征的统称，CDN 通过缓存键判断两个请求是否尝试加载同一资源。一般缓存键包括：

  • 请求行（方法 + 路径 + 查询参数）
  • Host 头
  • Vary 头指定的字段

当两个请求的缓存键相同时，CDN 直接返回缓存的响应，不再回源。

关键响应头

Cache-Control 常见值：

  • no-store：不得存储任何关于请求和响应的内容，每次都下载完整响应
  • no-cache：每次请求都需向服务器验证缓存是否过期（过期返回新内容，未过期返回 304）
  • public：响应可被任何中间人（CDN、代理等）缓存
  • private：响应仅用于单个用户，中间人不能缓存
  • max-age=<seconds>：资源能被缓存的最大时间

缓存投毒的必要条件

缓存投毒要成功，必须同时满足以下条件：

  1. 存在非缓存键能影响响应内容 — 某个请求头不在缓存键中，但服务端会将其值反射到响应里
  2. 响应可被缓存 — Cache-Control 未设置为 no-store，或 CDN 未严格遵循该指令
  3. 缓存键范围足够宽 — Vary 头覆盖字段少，使得大量正常用户会命中投毒缓存

三个条件缺一不可。如果非缓存键不影响响应内容，则无法注入恶意载荷；如果响应不可缓存，则投毒无法持久化；如果缓存键过于严格，则投毒只影响极少数用户。

通用利用步骤

  1. 探测非缓存键 — 用 Param Miner 或手动添加随机头，找出哪些头能影响页面内容但不参与缓存键计算
  2. 构造恶意响应 — 针对发现的非缓存键，构造能触发有害行为的请求
  3. 让投毒响应入缓存 — 发送构造好的请求，CDN 缓存恶意响应，后续用户全部受害

经典案例 — X-Forwarded-Host 投毒

X-Forwarded-Host 不在缓存键中，Cache-Control: public 允许 CDN 缓存 → 全站用户触发 XSS。

修复要点

严格设置缓存控制头

⚠️ 注意：即使设置了 no-store，CDN 也可能不完全遵循，所以 Vary 头和缓存键配置才是根本防线。

使用 Vary 头防止缓存复用

将影响响应内容的请求头纳入缓存键，确保不同输入不会共享同一份缓存。

CDN 配置缓存键

  1. 在 CDN 控制台中设置缓存键包含 Query String、User-Agent 等
  2. 设置缓存键排除动态参数（如 id, token）

设置缓存刷新策略

  1. 定期刷新缓存，防止恶意内容长期存在
  2. 对高风险页面设置较短缓存时间（如 5 分钟）
